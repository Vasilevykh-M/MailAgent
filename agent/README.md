# Mail agent

`mail-agent` — непрерывный worker на Python 3.11+, который находит **все** непрочитанные письма Яндекс Почты, анализирует тело и вложения, создаёт русскую сводку и добавляет либо обновляет строку `.xlsx` на Яндекс Диске.

Подробные публичные контракты уже существующих SDK и HTTP-сервисов приведены в [docs/public-contracts.md](docs/public-contracts.md). Исходный код `yandex/mail`, `yandex/drive`, `llm-service` и `ocr-service` агент не изменяет.

## Поток обработки

```text
all unread pages → oldest first → check idempotency → fetch(mark_read=False)
→ normalize body → metadata/plan/process attachments → OCR correction
→ Russian summary → Excel upsert/verification → table_committed → mark \Seen → completed
```

Поток одного письма реализован typed `LangGraph StateGraph`: `check_idempotency`, `fetch_message`, `normalize_message`, `collect_attachment_metadata`, `plan_attachments`, `process_attachments`, `validate_extractions`, `summarize_message`, `prepare_table_record`, `update_yandex_disk_table`, `commit_processing_state`, `mark_message_as_read`, `complete`. Ошибки ведут в отдельный узел `failure`. SQLite checkpointer LangGraph и собственная таблица состояния лежат по настраиваемым путям.

Для vLLM агент передаёт ограниченный общий объём текста: по умолчанию до `8000` символов для тела и всех извлечений из вложений, а также до `1200` токенов обычного ответа. Для `OCRCorrection` выделяется отдельный бюджет `3000` токенов, чтобы длинный исправленный текст не был оборван до закрытия JSON. Текст вложения длиннее `limits.chunk_size` делится на фрагменты, каждый фрагмент суммаризируется, а результаты объединяются в отдельную сводку вложения. По умолчанию анализируется до `24` фрагментов на вложение; при превышении лимита агент явно фиксирует предупреждение. Ошибка одного вложения не останавливает другие: письмо и все доступные уровни пересылки суммаризируются дальше, а в таблице у проблемного файла появляется отметка о ручной проверке. Для итоговой сводки выполняется до `llm.final_summary_attempts` полных попыток, затем агент пробует короткий контракт сводки по тексту письма без содержимого проблемных вложений. Полный текст остаётся в Markdown-отчёте. Запрос с HTTP `4xx` не повторяется автоматически, поскольку повторение не изменит его параметры.

Перед каждым запросом агент передаёт LLM доверенную текущую дату и время в UTC. В промптах явно разделены время обработки и дата письма: относительные сроки нормализуются только при однозначной точке отсчёта. Агент не выдаёт изменчивые мировые факты за проверенные без источника в письме или вложении.

`once` всегда листает Mail SDK постранично до `has_more=false`; `MAIL_BATCH_SIZE` задаёт размер страницы, а не предел обработки. Письма сортируются от старых к новым. `worker` сразу повторяет запрос после непустой очереди и ждёт `POLL_INTERVAL_SECONDS` только после пустой проверки. Отдельная ошибка записывается в SQLite и не останавливает остальные письма.

## Идемпотентность и Excel

`record_id` — SHA-256 от `mailbox`, UID и при наличии `Message-ID`. SQLite хранит стадии, число попыток, время следующей попытки, ошибки, checkpoint, хэши вложений и ключи кэша. Запись с `table_committed` после рестарта не проходит дорогой анализ: worker повторяет только установку `\Seen`, затем ставит `completed`.

XLS/XLSX разбираются локально как таблицы: агент передаёт в LLM названия листов, заголовки и подписанные значения строк, а не плоский TSV. Формулы и макросы не выполняются; для XLSX используются только сохранённые Excel результаты формул. Если таблица повреждена, пуста или содержит явно нечитаемые символы, файл не суммаризируется догадками: в сводке вложений и предупреждениях появляется название файла и сообщение о необходимости ручной проверки.

Если письмо уже получено, но дальнейшая нормализация, работа со вложением, OCR/VLM или LLM не завершилась, агент не оставляет его в `permanent_error`. Он сохраняет в таблице минимальную запись с итогом «Письмо требует ручной проверки», отмечает причину без текста письма и показывает её в панели наблюдения. Ошибки получения письма и сохранения на Диск остаются повторяемыми: без них нельзя безопасно создать запись или подтвердить её доставку.

Excel-обновление серийно в пределах процесса: агент читает метаданные и книгу во временный каталог, ищет строку по `record_id`, меняет или добавляет её, повторно сверяет удалённые метаданные, загружает с `overwrite=True`, скачивает окончательную версию и проверяет строку. Существующие листы, формулы, форматирование и параметры книги не пересоздаются. Конфликт повторяется ограниченно. У API Диска нет условной загрузки ETag, поэтому для одной книги рекомендован один core worker. Второй core worker с той же SQLite БД блокируется lock-файлом.

В листе Excel видны только отправитель, дата и тема письма, итоговая суммаризация, суммаризация вложений, а также ключевые факты и особенности — без JSON-обёрток. Служебный идентификатор строки скрыт. Значения длиннее лимита ячейки помечаются как сокращённые; полный Markdown-отчёт с разделами «Итоговая суммаризация», «Вложения» и «Содержимое письма» загружается в `report_directory`. Если создание книги отключено и файла нет, письмо остаётся непрочитанным с контролируемой ошибкой.

Для пересылки (`Forwarded message`, `Пересланное`/`Пересылаемое сообщение`, `Original message`) агент детерминированно разбирает все вложенные уровни, включая формат конверта Яндекс Почты с датой и отправителем в одной строке. Каждый уровень и внешний комментарий суммаризируются отдельно и затем объединяются; таблица использует реквизиты первого пересланного письма. Типовые конфиденциальные footer-блоки не участвуют в суммаризации.

## Вложения и безопасность

Имена вложений очищаются от traversal/контрольных символов и получают суффикс SHA-256; работа идёт в случайном временном каталоге, удаляемом в `finally`. Проверяется сигнатура, размер и тип. Поддержаны TXT, Markdown, HTML, JSON, XML, CSV, TSV, DOC/DOCX, XLS/XLSX, PPTX и PDF с текстовым слоем. Legacy DOC извлекается локально через системный `textutil` на macOS или `antiword`, если он установлен; Microsoft Office, макросы и внешний просмотрщик не запускаются. Если такого конвертера нет или файл повреждён, письмо продолжает обрабатываться, а в таблицу добавляется отметка о ручной проверке файла. Формулы не исполняются. Для неподдерживаемых, повреждённых и превышающих лимиты файлов сохраняются метаданные и предупреждение.

Перед обработкой LLM получает только безопасные признаки и строгую JSON Schema, сформированную из Pydantic-контракта; ответ валидируется до перехода к следующей стадии. JSON, вложенный моделью в Markdown или reasoning-обёртку, извлекается без журналирования его содержимого. При нарушении контракта журналируется только тип ошибки, причина завершения и число ошибок валидации, без ответа модели или пользовательских данных. LLM возвращает валидируемый JSON плана `programmatic`, `vision`, `ocr` либо `skip`. Решение сверяется с лимитами и `/api/v1/capabilities`; неверный выбор заменяется безопасным детерминированным fallback. Vision использует только локальные Data URL JPEG/PNG/WebP. OCR принимает файлы через multipart; низкая уверенность и артефакты запускают консервативную LLM-коррекцию, которая запрещает добавление фактов и использует маркеры `[unreadable]`.

Если OCR-сервис исчерпал повторы или нарушил HTTP-контракт, агент при `ocr.fallback_to_vlm=true` не останавливает письмо: PDF и изображения локально преобразуются в ограниченное число JPEG/PNG и передаются VLM только как Data URL. В результате сохраняется предупреждение, а `processing_tool` меняется на `vision`. Для форматов, которые нельзя безопасно представить изображением, или при ошибке VLM сохраняется обычная контролируемая ошибка без ложного успешного статуса.

Промпты на английском прямо запрещают выполнение инструкций из писем, документов и ссылок, запросы секретов и изменение правил. В итоговом результате нет hidden chain-of-thought: сохраняются только `tool`, `confidence`, краткая причина и предупреждения. Пользовательская сводка, факты, действия и ошибки — на русском.

## Установка и OAuth

```bash
cp agent/config.example.yaml agent/config.yaml
cp agent/.env.example .env
make install PROFILE=core
make auth-mail
make auth-drive
```

В `MAIL_ENV_FILE` и `DRIVE_ENV_FILE` укажите локальные `.env` SDK. Секреты находятся только в них или в окружении; YAML не содержит секретов. Приоритет: environment variables → YAML → безопасные defaults. Полный пример переменных есть в `.env.example`.

Основные команды:

```bash
mail-agent doctor
mail-agent once
mail-agent worker
mail-agent process --uid 12345 --mailbox INBOX
mail-agent process --uid 12345 --mailbox INBOX --reprocess
mail-agent retry-failed
mail-agent retry-failed --include-permanent
mail-agent show-state --uid 12345 --mailbox INBOX
mail-agent list-errors
```

`doctor` проверяет конфигурацию, каталоги, SQLite, импорты SDK, авторизацию Mail/Drive, endpoints LLM/OCR, capabilities/models, книгу и возможность захватить core-lock. Он не печатает секреты или содержимое письма.

## Панель наблюдения

```bash
make dashboard
```

Откройте `http://127.0.0.1:8765`. Панель работает параллельно с `make worker`, только читает SQLite и автоматически обновляется каждые три секунды. Она показывает состояние worker, время и результат последнего опроса, обрабатываемое письмо, текущий этап, очередь, последние успешные обработки и ошибки. Для идентификации письма сохраняются только отправитель, тема, дата, папка и UID; тело письма, вложения, OCR-текст, ответ LLM и секреты в панель не попадают. Сервер намеренно принимает соединения только с loopback-адреса; не публикуйте его через reverse proxy или VPN.

## Журнал работы

Все события выводятся в stdout в JSON. Для одного письма видны `message_processing_started`, `graph_run_started`, пары `stage_started`/`stage_completed`, обращения `llm_*` и `ocr_*`, обновление книги и итоговый `message_processed`. У каждого события есть технические идентификаторы письма, стадия, статус, попытка и `duration_ms`; для LLM/OCR также выводятся HTTP-статус, размер запроса и имя схемы или задачи.

```json
{"event":"stage_completed","component":"graph","stage":"summarize_message","status":"summarized","duration_ms":4821}
```

Логи намеренно не содержат тему, адреса, текст письма, имена и содержимое вложений, ответ LLM, OCR-результаты, JSON итогов, URL Диска или секреты. `httpx`-журнал подавлен на уровне INFO, поскольку его заменяют структурированные события клиентов.

## Makefile и развёртывание

```bash
make help
make install PROFILE=core|ai|all
make health PROFILE=core|ai|all
make once
make worker
make dashboard
make process UID=12345 MAILBOX=INBOX
make retry-failed
make start PROFILE=ai
make stop PROFILE=ai
make status PROFILE=ai
make test && make lint && make typecheck
```

`core` — неразделимый процесс `agent + Mail SDK + Drive SDK`. `ai` запускает существующий `llm-service/scripts/start.sh` и OCR через поддерживаемый `uvicorn app.main:app`; отдельных Mail/Drive процессов нет. В локальном режиме всё работает на одном устройстве. В двух устройствах core остаётся на A, а LLM+OCR на B; в трёх — LLM на B и OCR на C. Везде задайте `LLM_BASE_URL`/`OCR_BASE_URL`; файлы не разделяются: изображения идут в vLLM Data URL, документы в OCR multipart.

При удалённом LLM обязательно задайте API key, firewall/VPN или TLS reverse proxy. OCR без встроенной авторизации размещайте только в приватной сети, за reverse proxy или VPN. Не публикуйте API vLLM/OCR напрямую.

## Повторы, восстановление и завершение

HTTP 429/5xx, timeout и временные сетевые ошибки получают bounded exponential backoff с jitter. Permanent errors не повторяются бесконечно; `retry-failed --include-permanent` явно сбрасывает статус ошибки и запускает ручную попытку заново. Для уже готового или упавшего письма используйте только явную команду `process --uid … --mailbox … --reprocess`: она сбрасывает локальный checkpoint и повторяет полный анализ, обновляя строку с тем же `record_id`. Перед `\Seen` всегда происходит подтверждённая загрузка Excel и локальный `table_committed`; при любой ранней ошибке письмо непрочитано.

SIGINT/SIGTERM останавливает приём новых писем, сохраняет checkpoint после текущей атомарной операции, не ставит `\Seen` незавершённому письму, освобождает lock и закрывает HTTP-клиенты. Обычные тесты используют fakes/mocks и не выполняют OAuth, Yandex API, GPU, LLM или PaddleOCR. Реальные smoke-проверки должны быть отдельными opt-in тестами с настроенными endpoints.
