# Mail Agent

`Mail Agent` — локальный или серверный worker для обработки непрочитанных писем
Яндекс Почты. Он получает письмо и поддерживаемые вложения, извлекает из них
текст, при необходимости использует OCR и LLM, создаёт русскую сводку и
добавляет либо обновляет соответствующую строку в книге Excel на Яндекс Диске.

Проект разделён на самостоятельные компоненты. Агент использует SDK Почты и
Диска как Python-библиотеки, а сервисы LLM и OCR — по HTTP. В репозитории нет
общего Docker Compose, прокси или обязательного общего процесса: компоненты
можно размещать на одном компьютере либо на разных хостах.

## Возможности

- обработка всех непрочитанных писем выбранного ящика от старых к новым;
- безопасная работа с телом письма, пересылками и вложениями;
- извлечение текста из TXT, Markdown, HTML, JSON, XML, CSV, TSV, DOC/DOCX,
  XLS/XLSX, PPTX и PDF с текстовым слоем;
- OCR изображений и PDF через отдельный PaddleOCR-сервис, с контролируемым
  fallback на vision-модель для PDF и изображений;
- структурированная русская сводка и список действий без раскрытия внутреннего
  рассуждения модели;
- идемпотентное обновление таблицы Excel на Яндекс Диске по стабильному
  `record_id`;
- SQLite-состояние, возобновление после перезапуска, ограниченные повторы и
  отдельная обработка постоянных ошибок;
- локальная панель наблюдения и структурированные JSON-логи без содержимого
  писем, вложений, токенов и ответов модели.

## Состав репозитория

| Каталог | Назначение | Как взаимодействует с агентом |
| --- | --- | --- |
| [`agent/`](agent/README.md) | Основной worker `mail-agent`, LangGraph-процесс, SQLite, Excel-экспорт и панель наблюдения. | Запускается как один core-процесс. |
| [`yandex/mail/`](yandex/mail/README.md) | SDK и CLI для Яндекс Почты: OAuth 2.0, IMAP/XOAUTH2, MIME и вложения. | Импортируется агентом как `yandex-mail-sdk`. |
| [`yandex/drive/`](yandex/drive/README.md) | SDK и CLI для Яндекс Диска: OAuth 2.0, загрузка, выгрузка и метаданные файлов. | Импортируется агентом как `yandex-drive-sdk`. |
| [`ocr-service/`](ocr-service/README.md) | FastAPI-сервис PaddleOCR для OCR и структурного разбора документов. | Доступен агенту по HTTP, обычно на `:8000`. |
| [`llm-service/`](llm-service/README.md) | vLLM-сервис Qwen3.6 с Ray на двух GPU-узлах. | Доступен агенту через OpenAI-совместимый HTTP API, обычно на `:8001/v1`. |

Все компоненты имеют собственные зависимости, документацию и границы
ответственности. Не соединяйте сервисы внутренними импортами и не добавляйте
общие рантайм-каталоги между ними.

## Как устроена обработка

```text
Яндекс Почта
    │ непрочитанные письма, постранично
    ▼
mail-agent (core worker)
    │  ├─ нормализация тела и пересылок
    │  ├─ безопасная проверка и извлечение вложений
    │  ├─ SQLite: стадии, повторы, идемпотентность
    │  └─ русская структурированная сводка
    ├──────────────► OCR Service ──► PaddleOCR
    ├──────────────► LLM Service ──► Qwen через vLLM
    ▼
Яндекс Диск: Excel-реестр и Markdown-отчёты
    ▼
письмо отмечается прочитанным только после подтверждённой записи
```

Письмо проходит следующие основные стадии:

1. Агент получает страницу непрочитанных писем и проверяет локальное состояние
   по идентификатору, зависящему от ящика, UID и при наличии `Message-ID`.
2. Он читает письмо без установки флага `\Seen`, очищает и нормализует тело,
   рекурсивно разбирает пересланные сообщения и собирает метаданные вложений.
3. Для каждого вложения выбирается безопасный способ обработки: локальное
   программное извлечение, OCR, vision или пропуск с предупреждением.
4. Для ответов LLM используется строгая Pydantic/JSON-схема. Невалидный ответ
   не становится успешным результатом; применяется безопасный fallback либо
   фиксируется контролируемая ошибка.
5. Готовая сводка вносится в указанную книгу. Перед установкой `\Seen` агент
   повторно скачивает итоговый Excel-файл и проверяет добавленную строку.
6. После подтверждения записи состояние фиксируется как `table_committed`,
   письмо отмечается прочитанным и переводится в `completed`.

Если процесс остановлен после записи в таблицу, но до установки `\Seen`, при
следующем запуске дорогой анализ не повторяется: агент завершает только
безопасную операцию отметки письма.

## Требования

Для core-агента и SDK нужны:

- macOS или Linux;
- Python 3.11 или новее;
- [`uv`](https://docs.astral.sh/uv/);
- `make`;
- учётная запись Яндекс Почты и OAuth-приложения для Почты и Диска.

OCR-сервис по умолчанию работает на CPU. Для постоянной обработки больших
документов рекомендуется отдельный GPU-хост. LLM-сервис рассчитан на два
Linux-сервера с RTX 5090 и управляется исключительно своим `Makefile`;
подробные требования приведены в [его документации](llm-service/README.md).

`mail-agent` может обращаться к удалённым OCR и LLM. Для этого core-агент не
нужно размещать на GPU-машине: достаточно сетевой доступности заданных URL.

## Быстрый старт

Команды ниже выполняются из корня репозитория.

### 1. Подготовьте конфигурацию

Создайте локальные конфигурационные файлы из шаблонов:

```bash
cp agent/config.example.yaml agent/config.yaml
cp agent/.env.example .env
cp yandex/mail/.env.example yandex/mail/.env
cp yandex/drive/.env.example yandex/drive/.env
```

Файлы `.env`, `agent/config.yaml`, SQLite-базы, локальные отчёты и каталоги
`var/` являются рабочими данными. Не добавляйте в Git OAuth-secret, токены,
API-ключи или пользовательские документы.

Заполните как минимум OAuth-параметры в `yandex/mail/.env` и
`yandex/drive/.env`. В `.env` корня задаются параметры, которыми можно
переопределить значения `agent/config.yaml`; полный список находится в
[`agent/.env.example`](agent/.env.example).

### 2. Установите core-профиль

```bash
make install PROFILE=core
```

Команда создаёт изолированное окружение проекта `agent/`, устанавливает его
зависимости из lock-файла и подключает локальные SDK Почты и Диска как
редактируемые зависимости. Она не запускает OCR-модели, vLLM или OAuth.

### 3. Получите OAuth-токены

```bash
make auth-mail
make auth-drive
```

Команды запускают интерактивный Authorization Code Flow. Для Почты требуется
разрешение `mail:imap_full`, поскольку агент после успешной обработки меняет
флаг `\Seen`; для Диска нужны `cloud_api:disk.read` и `cloud_api:disk.write`.
Redirect URI обоих приложений должен быть:

```text
https://oauth.yandex.ru/verification_code
```

Токены сохраняются SDK локально и не должны публиковаться. Повторная
авторизация возможна через `yandex-mail auth --force` или соответствующую
команду SDK Диска.

### 4. Укажите доступные AI-сервисы

В `agent/config.yaml` либо в `.env` задайте адреса:

```yaml
llm:
  base_url: http://127.0.0.1:8001/v1
ocr:
  base_url: http://127.0.0.1:8000
```

Если сервисы развёрнуты на удалённых машинах, укажите их защищённые адреса в
`LLM_BASE_URL` и `OCR_BASE_URL`. Для удалённого vLLM обязательно настройте
`LLM_API_KEY`, TLS или закрытую сеть. OCR не имеет встроенной аутентификации,
поэтому размещайте его только в приватной сети, за VPN или reverse proxy с
аутентификацией.

### 5. Проверьте готовность и обработайте письмо

```bash
make health PROFILE=core
make health PROFILE=ai
make once
```

`make once` обходит весь ящик постранично, а не только одну страницу. Он
завершается после обработки доступных непрочитанных писем. При первой реальной
задаче OCR может скачать нужные модели в свой каталог кэша.

## Режимы запуска

| Задача | Команда | Результат |
| --- | --- | --- |
| Установить core | `make install PROFILE=core` | Агент и локальные SDK. |
| Установить OCR-зависимости | `make install PROFILE=ai` | OCR-сервис; LLM зависит от отдельной GPU-конфигурации. |
| Установить всё необходимое | `make install PROFILE=all` | Core и OCR-зависимости. |
| Проверить core | `make health PROFILE=core` | Конфигурация, SQLite, OAuth, endpoints и доступ к книге. |
| Проверить AI endpoints | `make health PROFILE=ai` | `GET /health` LLM и `GET /health/ready` OCR. |
| Однократный проход | `make once` | Обрабатывает все доступные непрочитанные письма. |
| Непрерывная работа | `make worker` | Опрос ящика с интервалом `POLL_INTERVAL_SECONDS`. |
| Запустить локальные AI-сервисы | `make start PROFILE=ai` | Запускает vLLM и OCR согласно их конфигурациям. |
| Запустить AI и worker | `make run PROFILE=all` | Сначала AI-сервисы, затем worker. |
| Остановить AI-сервисы | `make stop PROFILE=ai` | Останавливает OCR и vLLM. |
| Проверить статус AI | `make status PROFILE=ai` | Статус управляемых сервисов. |
| Локальная панель | `make dashboard` | Web-интерфейс на `http://127.0.0.1:8765`. |

`make run PROFILE=core` запускает только worker. Профиль `ai` предполагает,
что конфигурация `llm-service/` уже подготовлена на подходящей GPU-топологии;
см. [инструкцию для LLM-сервиса](llm-service/README.md).

Для разовой ручной обработки и восстановления доступны команды:

```bash
uv run --project agent mail-agent doctor
uv run --project agent mail-agent process --uid 12345 --mailbox INBOX
uv run --project agent mail-agent process --uid 12345 --mailbox INBOX --reprocess
uv run --project agent mail-agent retry-failed
uv run --project agent mail-agent retry-failed --include-permanent
uv run --project agent mail-agent show-state --uid 12345 --mailbox INBOX
uv run --project agent mail-agent list-errors
```

`--reprocess` сбрасывает checkpoint выбранного письма и повторяет полный
анализ. `--include-permanent` следует использовать только осознанно после
исправления причины ошибки: иначе постоянная ошибка будет повторена снова.

## Основная конфигурация агента

`agent/config.yaml` хранит несекретные настройки. Переменные окружения имеют
приоритет над YAML, а затем применяются безопасные значения по умолчанию.

| Раздел | Важные параметры | Назначение |
| --- | --- | --- |
| Пути | `work_dir`, `db_path`, `checkpoint_db_path` | Рабочий каталог, состояние обработки и checkpoint LangGraph. |
| Почта | `mail.mailbox`, `poll_interval_seconds`, `batch_size`, `mark_read_after_success` | Ящик, частота опроса, размер страницы и установка `\Seen`. |
| LLM | `base_url`, `model`, `max_text_chars_per_request`, `max_completion_tokens` | Адрес модели и ограничения объёма запросов. |
| OCR | `base_url`, `fallback_to_vlm` | Адрес OCR и возможность fallback на vision-модель. |
| Таблица | `remote_path`, `sheet_name`, `create_if_missing`, `report_directory` | Книга на Диске, лист и место для полных отчётов. |
| Панель | `host`, `port`, `queue_limit`, `recent_limit` | Локальная панель наблюдения и объём показываемой истории. |

Стартовые значения определяют:

- SQLite-файлы в `./var/mail-agent/`;
- почтовый ящик `INBOX`;
- опрос раз в 60 секунд;
- размер страницы 50 писем и последовательную обработку одного письма;
- Excel-файл `/mail-agent/mail-register.xlsx` и лист `Письма`;
- локальную панель только на `127.0.0.1:8765`.

В YAML не храните `LLM_API_KEY`, OAuth client secret, access token или refresh
token. Эти значения передавайте через локальные `.env` либо окружение процесса.

## Данные, идемпотентность и отчёты

В `work_dir` агент хранит только служебное состояние:

- `state.sqlite3` — стадия обработки, попытки, следующая попытка, хэши и
  безопасные диагностические сведения;
- `checkpoints.sqlite3` — checkpoint графа обработки;
- lock-файл, не позволяющий двум core-worker одновременно писать в одну и ту же
  SQLite-базу;
- полные Markdown-отчёты, если `table.save_full_report=true`.

Excel-реестр содержит ориентированные на пользователя сведения: отправителя,
дату, тему, итоговую сводку, сводку вложений, ключевые факты и особенности.
Служебный `record_id` скрывается. Агент не пересоздаёт существующие листы,
формулы, стили и параметры книги.

Один core-worker рекомендуется для каждой целевой книги: API Яндекс Диска не
даёт условной загрузки по ETag. Перед перезаписью агент ограниченно повторяет
проверку конфликта, повторно скачивает загруженную книгу и сверяет свою строку.

## Безопасность и приватность

- Почта читается без `\Seen`; флаг меняется лишь после проверенной записи в
  Excel и фиксации состояния.
- Имена вложений очищаются от traversal и управляющих символов. Временные
  файлы создаются в случайных каталогах и удаляются после обработки.
- Перед анализом проверяются размер, MIME-тип, расширение и сигнатура файла.
  Макросы, формулы, Microsoft Office и внешний просмотрщик не запускаются.
- Вложения и письма считаются недоверенным вводом. Промпты запрещают выполнять
  инструкции из документов, запрашивать секреты и менять системные правила.
- Логи содержат только технические события, стадии, длительности и безопасные
  причины ошибок; тела писем, данные вложений, OCR-текст, LLM-ответы, URL
  Диска, токены и секреты в них не пишутся.
- Панель наблюдения слушает только loopback-интерфейс. Не публикуйте её в
  интернете и не пробрасывайте без дополнительной защиты.
- Удалённые LLM/OCR endpoints должны быть доступны только доверенному агенту.
  Передаваемые документы могут содержать персональные или коммерческие данные.

## Поддерживаемые вложения и ограничения

Локально обрабатываются текстовые форматы, Office Open XML, XLS/XLSX, PDF с
текстовым слоем и ряд структурированных форматов. Legacy DOC извлекается через
`textutil` на macOS либо `antiword`, если он установлен. При отсутствии
конвертера, повреждённом файле, неподдерживаемом формате или превышенном лимите
письмо не теряется: сводка получает предупреждение для ручной проверки.

OCR API принимает JPEG, PNG и PDF. Его пределы — размер загрузки, число страниц
PDF, число одновременных инференсов и набор моделей — запрашиваются агентом из
`GET /api/v1/capabilities`; они не дублируются в логике worker. Подробности и
точные схемы описаны в [документации OCR-сервиса](ocr-service/docs/api.md).

У LLM ограничен общий объём текста, отправляемого в один запрос. Длинные
вложения делятся на фрагменты, каждый из которых суммаризируется отдельно; при
превышении допустимого числа фрагментов результат явно отмечается как
сокращённый. Это сохраняет предсказуемое потребление контекста и не создаёт
иллюзию полного прочтения документа.

## Наблюдение и диагностика

Все события выводятся в `stdout` как JSON. Полезные поля: `event`, `component`,
`stage`, `status`, `duration_ms`, технический идентификатор и номер попытки.
События `stage_started` и `stage_completed` позволяют отследить путь одного
письма, не раскрывая его содержимое.

Панель по адресу `http://127.0.0.1:8765` показывает:

- состояние worker и результат последнего опроса;
- обрабатываемое письмо и текущую стадию;
- очередь, недавние успешные обработки и ошибки;
- только отправителя, тему, дату, папку и UID — без текста и вложений.

Перед запуском worker используйте `make health PROFILE=core`. Команда `doctor`
проверяет конфигурацию, каталоги, SQLite, доступность SDK, авторизацию,
endpoints LLM/OCR, capabilities/models, книгу на Диске и возможность получить
lock. Она не печатает секреты.

## Развёртывание на нескольких хостах

Наиболее простой вариант — core, OCR и LLM на одном устройстве. Для рабочих
нагрузок компоненты можно разделить:

```text
Хост A: mail-agent + Яндекс OAuth + SQLite + доступ к Яндекс Диску
Хост B: vLLM API (LLM_BASE_URL для хоста A)
Хост C: OCR API  (OCR_BASE_URL для хоста A)
```

Файловая система между хостами не нужна: изображения отправляются в vLLM как
локальные Data URL, а документы в OCR — через multipart-запросы. SQLite и
рабочие отчёты остаются на хосте core-агента. Для LLM на двух GPU собственная
топология состоит из Ray head с API и Ray worker; её запуск, сетевые порты и
конфигурация подробно описаны в [`llm-service/README.md`](llm-service/README.md).

Не выставляйте Ray, NCCL, OCR или неаутентифицированный vLLM напрямую в
публичную сеть. Используйте VPN, private subnet, firewall и при необходимости
TLS reverse proxy.

## Проверки разработки

Быстрый набор проверок worker-агента:

```bash
make lint
make typecheck
make test
make check
```

Обычные тесты используют имитации внешних интеграций: они не запускают OAuth,
Yandex API, GPU, vLLM или реальную модель PaddleOCR. Реальные smoke-проверки
нужно выполнять явно после подготовки окружения. Для OCR предусмотрен opt-in
запуск `RUN_PADDLE_SMOKE=1`; для LLM используйте `make smoke` в
`llm-service/` только на корректно настроенной GPU-топологии.

Каждый сервис проверяется в своём каталоге и со своим lock-файлом. Команды и
ограничения приведены в соответствующих README; не запускайте ресурсоёмкие
проверки LLM на обычной рабочей станции.

## Типичные проблемы

| Симптом | Что проверить |
| --- | --- |
| `doctor` сообщает об ошибке авторизации | OAuth-параметры и срок действия токена в SDK `.env`; при необходимости повторите `make auth-mail` или `make auth-drive`. |
| Не проходит `make health PROFILE=ai` | Адреса `LLM_BASE_URL` и `OCR_BASE_URL`, сетевой доступ, запущенные сервисы и URL `/health`, `/health/ready`. |
| Письмо остаётся непрочитанным | Проверьте `mail-agent list-errors` и `show-state`; это ожидаемо до подтверждённой записи в Excel. |
| В таблице предупреждение по вложению | Файл может быть повреждён, неподдерживаем, превышать лимит, не иметь локального конвертера или не пройти OCR/LLM. |
| Второй worker не стартует | Один и тот же `work_dir` уже занят. Не запускайте два core-worker для одной SQLite-базы и одной книги. |
| OCR медленно отвечает при первом запросе | PaddleOCR лениво загружает модели. Сохраните каталог моделей в постоянном томе и оцените потребность в GPU. |

## Лицензирование и секреты

Перед развёртыванием проверьте условия использования Яндекс OAuth, Яндекс Диска,
PaddleOCR, PaddlePaddle, vLLM, Ray и выбранной модели. Не помещайте в репозиторий
реальные `.env`, токены OAuth, ключи LLM, model weights, документы пользователей
или созданные Excel-отчёты.
