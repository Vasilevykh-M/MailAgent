# Архитектура, безопасность и устранение проблем

## Состав проекта

```text
src/yandex_mail/
├── config.py        # загрузка .env и проверка параметров
├── oauth.py         # Authorization Code Flow и refresh grant
├── token_store.py   # OAuthToken и атомарное хранилище токенов
├── imap_client.py   # SSL/XOAUTH2-транспорт и UID-команды IMAP
├── parser.py        # RFC 5322/MIME-разбор
├── models.py        # публичные dataclass-модели
├── service.py       # высокоуровневый API
├── cli.py           # консольная команда
├── exceptions.py    # иерархия ошибок
└── utils.py         # безопасные файлы и вспомогательные функции
```

Разделение слоёв позволяет использовать сервис как библиотеку, независимо
тестировать HTTP/IMAP-взаимодействия и не смешивать MIME-разбор с транспортом.

## OAuth-поток

```text
authorize()
    │
    ├─ есть неистёкший токен? ── да ──> использовать его
    │
    └─ нет / истёк
          │
          ├─ есть refresh token? ── да ──> POST /token (refresh_token)
          │                                  │
          │                                  └─ сохранить новый токен атомарно
          │
          └─ refresh недоступен/отклонён ──> URL /authorize → код → POST /token
```

Authorization URL содержит `response_type=code`, Client ID, Redirect URI и
scope. Запросы к token endpoint отправляют Client Secret только по HTTPS через
`requests`; исключения и логи не включают секрет, verification code или токены.

## IMAP и XOAUTH2

Настройки подключения:

| Параметр | Значение |
| --- | --- |
| Сервер | `imap.yandex.com` |
| Порт | `993` |
| Шифрование | SSL/TLS |
| Механизм | `XOAUTH2` |

Формируется следующая **сырая** строка:

```text
user=<email>\x01auth=Bearer <access_token>\x01\x01
```

Её нельзя кодировать в Base64 вручную: `imaplib.authenticate()` делает это сам
после вызова callback. При неудаче аутентификации клиент обновляет access token
и пробует подключиться ещё раз. При второй неудаче выбрасывается
`ImapAuthenticationError`.

Публичные операции используют UID, а не sequence number. Sequence number может
измениться после появления или удаления письма, UID в рамках mailbox стабилен.

## Эффективность работы с почтовым ящиком

### Списки

Для `list_messages()` выполняются два этапа:

1. `UID SEARCH` находит соответствующие UID на сервере.
2. `UID FETCH` порциями получает `UID`, `FLAGS`, `RFC822.SIZE`, `INTERNALDATE`
   и выбранные заголовки.

При обычном списке полные MIME-тела не скачиваются. Для `get_all_messages()` и
`iter_messages()` один IMAP-сеанс обслуживает все пакеты. Генератор выдаёт
объекты по мере получения и не держит в памяти полный список метаданных.

### Полное письмо

`read_message()` запрашивает `BODY.PEEK[]`; это предотвращает неявную установку
`\Seen`. Если `mark_read=True`, SDK явно выполняет `UID STORE +FLAGS.SILENT
(\Seen)`, затем снова получает `FLAGS`. Таким образом, возвращённый объект
содержит подтверждённый сервером статус.

## MIME-разбор

`parser.py` использует `email.parser.BytesParser` со стандартной политикой
Python. Поддерживаются:

- `text/plain`, `text/html` и HTML-only сообщения;
- `multipart/alternative`, `multipart/mixed`, `multipart/related` и вложенные
  multipart-структуры;
- Base64 и quoted-printable;
- известные, отсутствующие и некорректные charset-декларации;
- RFC 2047-кодированные и Unicode-заголовки;
- inline-изображения, обычные вложения и сообщения без тела;
- повторяющиеся заголовки с сохранением порядка.

Текстовые части вложений не попадают в `text_plain` и `text_html`. Если
кодировка неизвестна, используется безопасный fallback UTF-8 с
`errors="replace"`; разбор не должен падать только из-за дефектного заголовка.

## Безопасность

### Секреты и токены

- Client Secret не хранится в исходном коде.
- `.env` и `.tokens.json` исключены из Git через `.gitignore`.
- Представление `OAuthToken` скрывает строковые токены в стандартном `repr`.
- Логи и тексты исключений не содержат Client Secret, access token, refresh
  token, verification code или полную XOAUTH2-строку.
- Диагностика сообщает лишь факт наличия и истечения токена.

### Файл токенов

`TokenStore.save()` создаёт временный файл в каталоге назначения, устанавливает
права `0600`, записывает JSON, вызывает `fsync()` и заменяет финальный файл
через `os.replace()`. Размещение временного файла в том же каталоге нужно для
атомарности замены на локальных файловых системах. На Windows управление правами
может осуществляться ACL, поэтому неудача `chmod` не ломает работу.

Повреждённый JSON не игнорируется молча: выбрасывается `TokenStorageError` без
вывода содержимого файла.

### Вложения и пути

Перед записью имя вложения:

1. приводится к basename;
2. очищается от `../`, `..\`, абсолютных путей и управляющих символов;
3. очищается от недопустимых символов Windows и зарезервированных имён;
4. проверяется, что итоговый путь остаётся внутри целевого каталога;
5. получает суффикс `_2`, `_3` и т. д. при конфликте;
6. записывается без перезаписи существующего файла, если она не разрешена явно.

Например, `../../passwords.txt` превратится в безопасный файл
`<выходной-каталог>/passwords.txt`.

## Ограничения и важные особенности IMAP

- `\Recent` зависит от сервера и может быть недоступен или иметь особенности
  семантики IMAP-сервера.
- Точный счётчик вложений в лёгком `MessageSummary` не вычисляется: для этого
  потребовалась бы дополнительная загрузка MIME-структуры каждого письма.
- Стандарт IMAP не определяет универсальный `SEARCH`-критерий «есть вложение».
  Используется серверный поиск по `Content-Type: multipart`, который эффективен,
  но является приближением.
- Сортировка списка выполняется по загруженным метаданным на клиенте, если
  сервер не предоставляет IMAP SORT.
- UID значим только в рамках выбранного mailbox. Нельзя использовать UID из
  `INBOX` для обращения к письму в другой папке.
- Статус `deleted` означает флаг `\Deleted`, а не обязательное физическое
  удаление письма с сервера.

## Диагностика проблем

### Не найден `.env` или конфигурация неполная

```bash
yandex-mail --env .env diagnose
```

Проверьте Client ID, Client Secret, email, Redirect URI и scope. Redirect URI и
scope должны соответствовать точным значениям из `.env.example`.

### OAuth не проходит

1. Убедитесь, что Redirect URI в настройках приложения —
   `https://oauth.yandex.ru/verification_code`.
2. Проверьте, что скопировали свежий verification code.
3. Проверьте Client ID и Client Secret.
4. Выполните `yandex-mail auth --force`.

### IMAP-аутентификация не проходит

1. Проверьте адрес в `YANDEX_EMAIL`.
2. Проверьте, включён ли IMAP-доступ для почтового аккаунта.
3. Повторите OAuth-авторизацию с `--force`.
4. Убедитесь, что scope — `mail:imap_full`.

### Письмо не найдено

Укажите правильный mailbox:

```bash
yandex-mail status 12345 --mailbox INBOX
```

Не используйте sequence number вместо UID. UID выводится первым столбцом команды
`yandex-mail list`.

### Вложения не найдены фильтром

`--has-attachments` — серверный эвристический фильтр по верхнему `Content-Type`.
Если письмо использует нестандартную MIME-структуру, найдите его по другим
критериям и выполните `read` или `attachments` для точного разбора.

## Логирование

Используется стандартный модуль `logging`.

| Режим | Назначение |
| --- | --- |
| Без флагов | Только предупреждения и понятные сообщения CLI |
| `--verbose` | Информационные сообщения |
| `--debug` | Отладочная информация и traceback при ошибке |

Даже в `--debug` SDK не должен логировать OAuth-токены, Client Secret,
verification code или полный XOAUTH2 payload.

## Тестирование

Тесты не подключаются к реальному Яндекс-аккаунту. HTTP-запросы OAuth и IMAP
клиент подменяются mock/fake-объектами.

```bash
pip install -e '.[dev]'
pytest
```

Набор тестов покрывает загрузку конфигурации, модель флагов, токен-файл,
OAuth URL и обмен кода, MIME-разбор, защиту путей вложений, XOAUTH2 без ручного
Base64, пакетные операции флагов и базовые сценарии CLI.
