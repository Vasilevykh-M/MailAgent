# Инструкции для работы с `frontend/`

## Область действия

Этот файл действует для всего дерева `frontend/`.

- Работайте только внутри `frontend/`, если пользователь отдельно не запросил
  интеграцию с другими сервисами.
- Не добавляйте импорты из `ocr-service/`, `llm-service/` или `yandex/`.
- Frontend — самостоятельное browser-приложение. Оно общается с Results API
  только по HTTP.
- Не редактируйте `node_modules/` и `dist/`: это локальные сгенерированные
  каталоги.

## Назначение приложения

`frontend/` — read-only dashboard для просмотра результатов обработки писем.
Основные возможности:

- аутентификация через пользовательскую Bearer-сессию;
- проверка состояния Results API;
- просмотр и фильтрация списка писем;
- просмотр статистики и графиков;
- открытие детальной карточки письма;
- скачивание вложений и исходного `.eml`.

Frontend не должен добавлять действия записи, которых нет в контракте API:
ответы на письма, переклассификацию, редактирование результатов или управление
пользователями.

## Технологии

- `React 19` и `TypeScript`;
- `Vite` для dev server и production build;
- `React Router` для маршрутов;
- `TanStack Query` для server state и кеша;
- `Zod` для runtime-валидации ответов API;
- `CSS Modules` и CSS-переменные для стилей;
- `Recharts` для графиков;
- `MSW` для dev-only моков API;
- `Vitest`, `Testing Library` и `jsdom` для тестов;
- `Oxlint` и `Oxfmt` для проверки кода.

Используйте `npm`: зависимости зафиксированы в `package-lock.json`.

## Архитектурные границы

Соблюдайте направление зависимостей:

```text
main -> app -> features -> api
                     \-> shared
              api -> shared/lib
```

- `src/app/` только собирает providers, routes и крупные feature-компоненты.
- `src/features/` содержит пользовательские сценарии и композицию UI.
- `src/api/` инкапсулирует HTTP, авторизацию, валидацию и TanStack Query.
- `src/shared/` не должен импортировать `app`, `features` или `api`.
- Компонент не должен выполнять произвольный `fetch`, если запрос можно выразить
  через `src/api/client.ts` и query hook.
- Не добавляйте второй глобальный state manager. Server state хранится в
  TanStack Query, локальное состояние интерфейса — в React.
- Выбранное письмо является частью URL (`/emails/:recordId`), а не только
  локального состояния.

Текущий поток загрузки данных:

1. `src/main.tsx` при необходимости запускает MSW.
2. `src/app/providers.tsx` подключает Query, Theme, Router и Auth providers.
3. `src/app/router.tsx` выбирает страницу и защищает приватные маршруты.
4. Feature вызывает hook из `src/api/queries.ts`.
5. Query hook вызывает типизированную функцию из `src/api/client.ts`.
6. Клиент добавляет Bearer header, обрабатывает HTTP-ошибки и валидирует JSON
   через Zod.
7. Feature получает только проверенные данные либо явное состояние ошибки.

## Где размещать изменения

### Новый API endpoint

Изменяйте согласованно:

1. `src/api/schemas.ts` — Zod-схема ответа;
2. `src/api/types.ts` — тип через `z.infer`, без ручного дублирования схемы;
3. `src/api/client.ts` — HTTP-функция;
4. `src/api/queryKeys.ts` — стабильный query key, если endpoint кешируется;
5. `src/api/queries.ts` — query hook;
6. `src/api/index.ts` — публичный экспорт;
7. `src/mocks/handlers.ts` и `src/mocks/data.ts` — dev-мок;
8. соседний `*.test.ts` или `*.test.tsx` — проверка преобразований и ошибок;
9. `api_reference.md` и `README.md` — если изменился внешний контракт или
   пользовательское поведение.

Не возвращайте в UI непроверенный результат `response.json()`. Ответы API должны
проходить через Zod на границе клиента.

### Новая страница или feature

- Размещайте пользовательский сценарий в `src/features/<feature>/`.
- Крупный компонент храните в отдельной директории:

```text
ComponentName/
  ComponentName.tsx
  ComponentName.module.css
  index.ts
```

- Добавляйте route в `src/app/router.tsx`.
- Экспортируйте feature через локальный `index.ts`, затем через
  `src/features/index.ts`.
- Общий компонент выносите в `src/shared/ui/` только после того, как он не
  зависит от бизнес-терминов конкретной feature.

### Новый общий UI-компонент

- Используйте структуру `src/shared/ui/ComponentName/`.
- Компонент должен принимать данные и callbacks через props, а не знать про
  Results API или TanStack Query.
- Добавьте экспорт в локальный `index.ts` и `src/shared/ui/index.ts`.
- Предусмотрите доступные имена, keyboard interaction и `focus-visible`.
- Явно показывайте состояния `loading`, `empty`, `error` и `disabled`, когда они
  применимы.

### Новая утилита или hook

- Чистые функции помещайте в `src/shared/lib/` и покрывайте unit-тестами рядом.
- Общие hooks помещайте в `src/shared/hooks/<hookName>/`.
- Доменные преобразования оставляйте рядом с feature, если они не нужны всему
  приложению.

## API, конфигурация и авторизация

- В dev режиме base URL берётся из `VITE_RESULTS_API_BASE_URL`.
- В preview/production base URL строится из protocol и hostname страницы с
  фиксированным port `8080`; dev-переменная там намеренно игнорируется.
- `VITE_DEFAULT_MAILBOX` задаёт mailbox по умолчанию.
- `VITE_ENABLE_API_MOCKS=false` отключает MSW в dev режиме.
- Любая переменная `VITE_*` попадает в browser bundle. Не храните там API keys,
  пароли, реальные токены и другие секреты.
- Access token хранится через `src/api/authToken.ts` в `localStorage` и
  передаётся как `Authorization: Bearer <token>`.
- Не логируйте токены, содержимое писем, headers, вложения, персональные данные
  и ответы API целиком.
- Downloads выполняются через `fetch` из `src/api/downloads.ts`, чтобы сохранить
  auth header и корректно обработать API error. Не заменяйте этот путь обычной
  ссылкой без учёта авторизации.
- Относительные download URL преобразуйте через общий helper, а не конкатенацией
  строк.

## Работа с моками

- MSW запускается только при `import.meta.env.DEV`.
- `src/mocks/handlers.ts` должен повторять HTTP status, поля, пагинацию,
  авторизацию и ошибки из реального контракта.
- `src/mocks/data.ts` содержит только вымышленные данные без реальной переписки
  и персональных данных.
- Не допускайте, чтобы production/preview зависели от моков.
- `public/mockServiceWorker.js` — сгенерированный MSW worker. Не редактируйте
  его вручную; обновляйте штатной командой MSW только при необходимости.
- `onUnhandledRequest: 'bypass'` оставляйте осознанно: неизвестные запросы в dev
  уходят в реальную сеть.

## UI и стили

- Стили компонента размещайте в соседнем `*.module.css`.
- Общие цвета, spacing, radii и размеры берите из CSS-переменных
  `src/index.css`.
- Поддерживайте обе темы через существующие токены; не хардкодьте цвет,
  подходящий только одной теме.
- Следуйте `design_system.md`: сначала читаемость данных, затем декоративность.
- Цвет статуса всегда дублируйте текстом или доступной подписью.
- Используйте существующие компоненты из `src/shared`, прежде чем создавать
  локальный аналог.
- Иконки берите из `lucide-react`; декоративным иконкам задавайте
  `aria-hidden="true"`.
- Не рендерите `original_email.text_html` через `dangerouslySetInnerHTML`.
  Содержимое писем считается недоверенным и отображается только как текст.
- Сохраняйте адаптивность от минимальной ширины `320px`.

## TypeScript и импорты

- Сохраняйте строгие типы и явную обработку nullable/optional полей API.
- Не используйте `any`, type assertions и non-null assertions без необходимости,
  которую нельзя выразить проверкой или схемой.
- Для type-only imports используйте `import type`.
- Публичные импорты между слоями делайте через их `index.ts`.
- Внутри одного компонента допустимы относительные импорты соседних файлов
  реализации и CSS module.
- Не создавайте циклические barrel imports. Если внутреннему модулю нужен
  конкретный соседний файл, импортируйте его напрямую.
- Не сортируйте импорты вручную вопреки Oxfmt: `npm run format:write` применяет
  установленный порядок.

## Тесты

- Unit-тесты размещайте рядом с кодом как `*.test.ts` или `*.test.tsx`.
- Проверяйте чистые преобразования, URL/config resolution, Zod-схемы, обработку
  ошибок, query-dependent поведение и значимые UI-состояния.
- Для UI используйте Testing Library и запросы по роли/доступному имени, а не по
  CSS-классам или внутренней структуре.
- Не выполняйте реальные сетевые запросы в unit-тестах.
- Изменение API-контракта должно иметь тест на валидный ответ и важные
  несовместимые/ошибочные варианты.
- Реальный backend не требуется для `npm run test`; MSW dev worker и Vitest —
  разные механизмы.

## Рабочие команды

Запускайте команды из `frontend/`:

```bash
cd frontend
npm ci
cp .env.example .env.local
npm run dev
```

Для изменения зависимостей используйте `npm install <package>` или
`npm uninstall <package>`, чтобы `package.json` и `package-lock.json` оставались
синхронизированы.

Проверки:

```bash
npm run format
npm run lint
npm run test
npm run build
```

- `npm run format:write` исправляет форматирование.
- `npm run test:watch` запускает Vitest в watch mode.
- `npm run preview` проверяет уже собранное приложение локально.
- Начинайте с наиболее узкой проверки изменённого модуля, затем выполняйте
  полный набор перед завершением работы.

## Документация

- Документацию и пользовательские тексты пишите на русском.
- Не переводите команды, пути, env names, JSON fields, model/API identifiers и
  реальные сообщения API.
- При изменении запуска, env или возможностей обновляйте `README.md`.
- При изменении API обновляйте `api_reference.md`.
- При изменении токенов или общих UI-правил обновляйте `design_system.md`.
- Не описывайте будущую функциональность как уже реализованную.

## Критерии готовности

Перед завершением:

1. Изменения находятся только в `frontend/`.
2. API и данные проходят через существующие архитектурные границы.
3. Добавлены или обновлены необходимые тесты и документация.
4. Выполнены `npm run format`, `npm run lint`, `npm run test` и `npm run build`,
   либо явно указано, почему конкретная проверка не запускалась.
5. Проверен Git diff, включая отсутствие изменений в защищённом `yandex/`.
6. В итоговом отчёте перечислены изменения, выполненные и невыполненные
   проверки, область работ и статус проверки `yandex/`.
