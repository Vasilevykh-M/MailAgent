# Дизайн-система dashboard

Документ фиксирует базовые визуальные решения для постепенной разработки
dashboard. Цель — простой, современный и лаконичный интерфейс без тяжёлого
UI-kit.

## Принципы

- Сначала читаемость данных, затем декоративность.
- Один главный акцентный цвет, без перегруженной палитры.
- Плотность умеренная: dashboard должен помещать много информации, но не
  выглядеть как таблица администрирования.
- Все состояния данных видимы: loading, empty, error, stale/refetching.
- Цветовые статусы всегда дублируются текстом.

## Визуальный Тон

- Умеренно тёмная аналитическая панель, без почти чёрных поверхностей.
- Карточки со спокойными границами и слабой прозрачностью.
- Акцент `cyan` для активных элементов, статуса доступности и primary actions.
- Семантические цвета только для статусов: success, warning, danger, info.

## Токены

Основные CSS-переменные находятся в `src/index.css`.

### Цвета

- `--color-bg` — фон приложения.
- `--color-surface` — обычная поверхность карточки.
- `--color-surface-muted` — полупрозрачная поверхность.
- `--color-surface-strong` — контрастная вложенная поверхность.
- `--color-border` — стандартная граница.
- `--color-border-strong` — усиленная граница.
- `--color-text` — основной текст.
- `--color-text-muted` — вторичный текст.
- `--color-text-subtle` — малозаметный текст.
- `--color-accent` — основной акцент.
- `--color-success` — успешное состояние.
- `--color-warning` — предупреждение.
- `--color-danger` — ошибка или ручная проверка.
- `--color-info` — информационный статус.

### Радиусы

- `--radius-sm` — бейджи, маленькие controls, около `6px`.
- `--radius-md` — inputs, buttons, вложенные panels, около `10px`.
- `--radius-lg` — карточки, около `14px`.
- `--radius-xl` — крупные sections, около `18px`.

Радиусы не должны съедать полезную площадь controls. Для input используется
`--control-padding-inline`, чтобы текст не был прижат к краям и первый символ не
терялся внутри скругления.

### Spacing

- `--space-1` … `--space-8` — базовая шкала `4px`.
- `--space-page-x`, `--space-page-y` — отступы страницы.

## Типографика

- Font stack: system UI.
- Главный заголовок: 32–48px.
- Заголовки карточек: 16–20px.
- Основной текст: 14–16px.
- Technical text: 12–13px, моноширинный только для ID/URL/hash.

## Базовые Компоненты

Компоненты хранятся в отдельных директориях:

```text
ComponentName/
  ComponentName.tsx
  ComponentName.module.css
  index.ts
```

Каждый слой имеет общий `index.ts` для реэкспорта. Приложение импортирует
компоненты через индекс слоя, например `src/shared/index.ts` или
`src/features/index.ts`, а не напрямую из файлов реализации.

### `Card`

Используется для KPI, фильтров, списка, detail blocks.

Варианты:

- `default` — обычная карточка.
- `muted` — менее контрастная карточка.
- `strong` — вложенный panel или technical block.

### `Badge`

Используется для статусов и коротких меток.

Варианты:

- `neutral`
- `accent`
- `success`
- `warning`
- `danger`
- `info`

### `Button`

Используется для обновления, загрузки следующей страницы, скачивания.

Варианты:

- `primary`
- `secondary`
- `ghost`
- `danger`

### `Alert`

Используется для ошибок API, предупреждений обработки, unavailable state.

Варианты:

- `info`
- `warning`
- `danger`
- `success`

### `EmptyState`

Используется для пустого списка, отсутствующей карточки, пустых вложений.

## Маппинг Статусов

### Backend health

- `ok` → `success`.
- `unavailable` → `warning`.
- ошибка запроса → `danger`.

### Classification status

- `classified` → `success`.
- `new_project` → `info`.
- `manual_review` → `warning` или `danger`, если есть критичные warnings.
- `null` → `neutral`.

### Confidence

- `>= 0.8` → `success`.
- `>= 0.5` и `< 0.8` → `warning`.
- `< 0.5` → `danger`.
- `null` → `neutral`.

## Layout MVP

Первый рабочий экран:

- Header с названием, состоянием API и кнопкой обновления.
- Панель фильтров.
- KPI row.
- Основная область `list + detail`.
- На узких экранах detail уходит ниже списка.

## Ограничения

- Не рендерить `original_email.text_html` как HTML.
- Не показывать длинные технические JSON по умолчанию.
- Не логировать содержимое писем, вложений, headers или ключи.
- Не добавлять frontend-only действия, которых нет в API.
